Tray reorganization — four directions

The current tray mixes four unrelated concerns in one row: what model is running, what mode it runs in, where it runs, and what tools it has. Each option below reorganizes those concerns using a different design lens — grouping, spatial separation, progressive disclosure, and density reduction — so we can compare tradeoffs side-by-side.

00 — REFERENCE Current state

Seven controls compete for attention in a single horizontal scan, with no grouping and no hierarchy between primary (model, submit) and secondary (saved, agents) actions. Workspace context (worktree + branch) is detached at the bottom, visually orphaned.

01 Grouped clusters with workspace breadcrumb

Apply Gestalt grouping: bundle related controls into segmented pill clusters so the eye reads three concepts (model, mode, tools) instead of six items. Workspace context becomes a breadcrumb above the input — it answers "where am I?" before "what am I asking?". Saved is removed from the tray and surfaces as a slash-command (/saved) since it's a navigation action, not a per-prompt setting.

02 Context header, action footer

A clear spatial split: everything that describes where you are (worktree, branch, agents) sits above the textarea; everything that controls how the turn runs (model, reasoning, mode, access) sits below. This mirrors how IDEs separate status bars from action bars. Saved moves to the global sidebar — it's a library, not a prompt-time choice.

03 Minimal default, expand for advanced

Progressive disclosure. 90% of prompts don't need the user to think about reasoning effort, sandbox mode, or permissions — those settings persist between turns. Default shows only model + submit + a "···" affordance. Click it to reveal the rest. A single-line status footer keeps worktree/branch visible at a glance without consuming tray real estate. Best choice if telemetry shows most users rarely change tray settings.

Cleanest default state Hides power-user controls until needed Risk: discoverability for new users

04 Icon-forward compact

Density via iconography. Once a setting is familiar, its label is noise — Linear, Raycast and Arc all rely on this. Reduces six labeled chips to a row of icons; only the model (the most identity-laden choice) keeps its label. The branch becomes a single monospace pill on the left, anchoring workspace context inside the action row. Best for high-frequency users; weaker for first-run learnability — pair with persistent tooltips.

Halves visual weight Branch + tray on one line Requires good tooltips

Round 02 — refined

Bookmark in the corner, branch stays at the bottom

Saved prompts belong in the input — but as a quiet bookmark icon pinned to the top-right corner, not a labeled chip in the action row. Branch info stays at the bottom. Tray reflects priority: model, reasoning, mode stay loud and labeled; context window and access level get demoted because they're rarely changed per turn. Four ways to express that hierarchy, ordered from least to most aggressive demotion.

08 Hide low-priority behind a sliders icon

Push the demotion all the way: context and access aren't visible by default. A sliders icon next to submit opens a popover with both. Best if you trust the priority ordering: it costs one extra click to change context or access, but recovers visual weight on every other turn.
